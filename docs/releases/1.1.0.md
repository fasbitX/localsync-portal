# Release 1.1.0

**Date:** 2026-04-15
**Type:** Minor - New Feature

## Summary

Local client is now the single source of truth. All local operations (delete photo, delete folder, rename folder) propagate to the remote server. The remote server mirrors whatever the local client tells it.

## New Feature: Full Sync Propagation

Previously, only file uploads and folder creation synced to the remote server. Deletes and renames were local-only, leaving stale data on the remote. Now every mutation propagates.

### Remote Server -- New Sync Endpoints

| Endpoint | Purpose |
|----------|---------|
| `DELETE /api/photos?relativePath=...` | Delete a photo by path (DB + disk) |
| `DELETE /api/folders?relativePath=...` | Delete a folder and all its photos (DB cascade + disk) |
| `PUT /api/folders/rename?oldPath=...&newPath=...` | Rename a folder, batch-update all photo paths, rename on disk |

All endpoints use API key authentication (`X-API-Key` header), same as upload. All are idempotent -- deleting something already gone returns 200, not an error.

### Local Client -- Sync Functions

Three new functions in `sync.js`:
- `syncDeletePhoto(relativePath)` -- calls remote delete photo endpoint
- `syncDeleteFolder(relativePath)` -- calls remote delete folder endpoint
- `syncRenameFolder(oldPath, newPath)` -- calls remote rename endpoint

All are best-effort: failures are logged but do not block the local operation. Includes a dedup mechanism (10-second TTL) to prevent duplicate remote calls when both the route handler and file watcher detect the same deletion.

### Local Client -- Route Wiring

Each local operation in `routes/folders.js` now triggers the corresponding remote sync:
- Delete folder → `sync.syncDeleteFolder()`
- Delete photo → `sync.syncDeletePhoto()`
- Rename folder → `sync.syncRenameFolder()`

### Local Client -- File Watcher

The file watcher (`watcher.js`) now listens for:
- `unlink` events (file deleted from disk) → cleans sync_log, calls `sync.syncDeletePhoto()`
- `unlinkDir` events (directory deleted from disk) → calls `sync.syncDeleteFolder()`

This means files deleted directly from the filesystem (not via the UI) also propagate to the remote server.

- **`watcher.js`** -- added `unlink`/`unlinkDir` event handlers
